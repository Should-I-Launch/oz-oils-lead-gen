# Migrations (Alembic)

Schema is managed entirely by Alembic. Autogenerate is the default workflow.

## Layout

```
backend/
├── alembic.ini
└── migrations/
    ├── env.py          # auto-imports every app.features.<name>.models
    ├── script.py.mako
    └── versions/
        └── 0001_initial.py
```

## How autogeneration sees your models

`migrations/env.py` walks `app/features/` and imports each `models.py` or `models/` package, so any `Base.metadata`-bound table is visible to `--autogenerate`. **You do not need to register new features manually.** Just create `app/features/<name>/models.py` or `app/features/<name>/models/` with classes inheriting from the shared `Base`.

## Workflow

### Create a migration

```bash
docker compose exec backend alembic revision --autogenerate -m "add <name>"
```

Inspect the generated file in `migrations/versions/` before applying — autogenerate is good, not perfect.

### Apply

```bash
docker compose exec backend alembic upgrade head
```

### Rollback one

```bash
docker compose exec backend alembic downgrade -1
```

### History / status

```bash
docker compose exec backend alembic history
docker compose exec backend alembic current
```

## Initial migration

`0001_initial.py` is empty (the boilerplate ships with no domain tables). Your first feature creates the first real migration.

## Rules

- **One migration per logical change.** Don't pile schema changes into one revision.
- **Inspect autogenerated migrations.** Especially renames — Alembic treats them as drop+add unless you intervene.
- **Migrations are append-only history.** Don't edit a migration after it's been applied in any environment. Write a new one.
- **DB extensions go in migrations**, not in a setup script. Use `op.execute('CREATE EXTENSION IF NOT EXISTS ...')`.

## CI / fresh DB

```bash
docker compose down -v
docker compose up -d db
docker compose up -d --build backend
docker compose exec backend alembic upgrade head
```
